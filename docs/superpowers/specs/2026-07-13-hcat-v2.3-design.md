# v2.3 — hcat: compress at the source (design)

Date: 2026-07-13 · Status: shipped in v2.3.0

## Problem

Every earlier layer of this plugin (badge, missed counter, Dangi) is *advisory
after the fact*. By the time Claude can call `mcp__headroom__headroom_compress`,
the big output is already in context — those input tokens are spent — and the
tool only accepts inline `content`, so Claude must re-emit the blob as
tool-call arguments (output tokens, ~5× the input price) while the raw
original stays in the transcript anyway. In practice `headroom_compress` pays
off inside subagents (compress before returning to the parent); in the main
session the measured "savings" are mostly illusory. Live session stats
confirmed it: 0 main-loop compressions, all real savings in the subagent
bucket.

## Decision

Add a prevention layer that keeps raw bytes out of context in the first place.
Two pieces, both shipped in `scripts/` and installed to `~/.claude/` by the
existing SKILL.md installer:

### hcat (`scripts/hcat`)

`hcat <file>` → compressed rendering on stdout, raw bytes never enter context.

- Bash arg-validation wrapper + embedded Python heredoc run under headroom's
  venv python. Resolution: `$HCAT_PYTHON` (authoritative override — no
  fallback if broken) → sibling `python` of `headroom` on PATH →
  `~/.headroom-venv/bin/python`.
- Calls `headroom.compress.compress([{"role":"tool","content":<file>}])` —
  the same entry point the MCP server's `_compress_content` uses. Measured:
  182 KB JSON → 70.1 % token reduction in ~0.6 s cold.
- **No retrieval store.** headroom's `CompressionStore` is in-memory
  per-process, so a hash minted by an `hcat` process would be unretrievable
  from the MCP server. Instead the header cites the original path: the file on
  disk is the source of truth; Read it with offset/limit for exact details.
- Passthrough when savings < 5 % (schema noise isn't worth it).
- Appends a `{"type":"compress","strategy":"hcat",...}` event to headroom's
  shared session-stats file (`headroom.paths.session_stats_path()`, flock,
  same schema as `SessionStats.record_compression`) so `headroom_stats`
  counts it. Best-effort — never fatal.
- Exit codes: 0 ok · 2 usage/unreadable · 3 python missing · 4 compress failed.

### hcat gate (`scripts/hcat-gate.sh`)

PreToolUse hook, matcher `Read`. Denies a Read of a ≥ `HCAT_GATE_BYTES`
(16384) file with extension `.json/.jsonl/.ndjson/.csv/.tsv/.log` — once per
file per session (state: `$STATE_DIR/session-<sid>.gate`, one path per line).
The deny reason contains the exact `hcat` command to run. Retry of the same
Read passes: a redirect, never a wall. Allows everything when headroom isn't
runnable (never deny toward a dead end). `HCAT_GATE_OFF=1` disables. Same
contract as dangi-hook: always exit 0, single JSON on stdout, silent stderr.

### Dangi changes

- Excludes `Edit|Write|MultiEdit|NotebookEdit` (they echo the code being
  changed — never compression targets; confirmed live false positives).
- Nudge text now routes: file-backed → `hcat`; otherwise →
  `headroom_compress` or a disposable subagent that returns compressed text.

## Known limitations (deliberate)

- The statusline badge counts only MCP `headroom_compress` transcript events,
  so hcat savings appear in `headroom_stats` (sub_agents/combined bucket) but
  not the badge. Undercounts, never overcounts. Candidate for v2.4.
- Gate fires by extension + size only; a huge structured `.txt` passes.
- Each hcat invocation logs one stats event; piping hcat twice = two events.

## Verification

- `test.sh`: 22 new checks (hcat contract, gate deny/escape-hatch/kill-switch,
  Dangi exclusions + new text); suite green.
- Live in-session: gate denied first Read of a 182 KB JSON with hcat guidance,
  retry passed; installed hcat compressed it 70.1 %; the event appeared in
  `headroom_stats`; Dangi's new text fired on a real Bash output; Dangi stayed
  silent on Edits.
- Payload-schema lesson from the v2.2.1 bug applied: gate tests build stdin
  with the real `tool_input.file_path` field names.
